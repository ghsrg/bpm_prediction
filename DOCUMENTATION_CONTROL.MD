# DOCUMENTATION_CONTROL.MD

## 1) Призначення документа
`DOCUMENTATION_CONTROL.MD` — це master-трекер робіт з документації для розробки ПЗ в межах дисертаційного дослідження.

Він одночасно фіксує:
- поточний стан документації;
- покроковий план оновлення файлів;
- буфер домовленостей команди (канон термінів, стек, правила оформлення);
- трасованість «теза дисертації → документ/артефакт реалізації».

---

## 2) Канон та рамки scope

### 2.1 Канон джерела
- Єдиний канон змісту: `Дисертація Коротенко.docx`.
- У разі конфлікту між `.MD` файлами та дисертацією, пріоритет має дисертація.

### 2.2 Статус TODO у дисертації
- `TODO` фрагменти дисертації вважаються **поза scope** поточного етапу розробки.
- Вони мають бути доопрацьовані після реалізації ПЗ та перед захистом.

### 2.3 Наукові задачі, які мають бути підтримані документацією
1. Оцінка впливу структурної обізнаності на якість інференсу (EPOKG, стійкість до деградації при зміні обсягу контексту).
2. Перевірка безперервної еволюції архітектури через латентне кондиціонування версій регламентів (мінімізація катастрофічного забування).
3. Валідація механізму Reliability Semaphore для детекції OOD станів (топологічний дрейф) та балансування структурної релаксації.
4. Валідація ефективності дуальної архітектури через мінімізацію метрики OOS порівняно з локально-оптимізованими моделями.
5. Комплексна експериментальна апробація наскрізної архітектури в динамічному середовищі з фіксацією обмежень та меж застосовності.

Результати цих задач мають бути формалізовані у розділі 4 дисертації: **«ЕКСПЕРИМЕНТАЛЬНЕ ДОСЛІДЖЕННЯ ТА ВАЛІДАЦІЯ МЕТОДОЛОГІЇ»**.

## 3) Буфер домовленостей (база правил)

### 3.1 Термінологія
- Підтримуються обидва терміни:
  - **OOD** — стан *Out-of-Distribution*;
  - **OOS** — метрика *Out-of-Scope Rate*.
- Терміни мають бути узгоджені між `GLOSSARY.MD`, `VARIABLES.MD`, `ARCHITECTURE*.MD`, `README.MD`.

### 3.2 Технологічні домовленості для реалізації
- Мова: **Python 3.10+**.
- Бібліотеки ядра: **PyTorch Geometric**, **pm4py**.
- Кодова база: з **Type Hints**.
- Коментарі у коді: **українською**, змістовно, без надмірного дублювання очевидного.

### 3.3 Підхід до розробки
- Основний процес: **Top-Down (Contract-First)** + **Iterative Enhancement**.
- Контракти, порти та інваріанти фіксуються до реалізації конкретних адаптерів/моделей.
- Кожна ітерація завершується перевіркою інваріантів, метрик та синхронізацією документації.

### 3.4 Архітектурні вимоги PoC (enterprise-ready)
- Цільовий стиль: **Clean Architecture** у поєднанні з патерном **Pipeline** для наукових експериментів.
- Інтеграційний принцип: **Гексагональна архітектура (Ports & Adapters)** для ізоляції доменного ML-ядра від інфраструктури.
- Варіативність алгоритмів забезпечується через **Strategy Pattern** (drift/loss/model config).

### 3.5 Дисципліна документації
- Усі нові/оновлені документи використовують терміни з `GLOSSARY.MD`.
- Позначення змінних, метрик, коефіцієнтів беруться з `VARIABLES.MD`.
- Перед створенням нових файлів (contracts/ABC/flows) необхідно вирівняти базові документи поточного контуру.

## 6) Definition of Done для хвилі оновлення поточних файлів
Хвиля вважається завершеною, якщо:
1. Оновлені `GLOSSARY.MD`, `VARIABLES.MD`, `ARCHITECTURE_GUIDELINES.MD`, `ARCHITECTURE.MD`, `README.MD`.
2. Між файлами немає термінологічних та символічних суперечностей.
3. Документація явно підтримує 5 дослідницьких задач зі вступу.
4. Після цього можна переходити до створення `Contracts & ABC` і опису потоків даних.
