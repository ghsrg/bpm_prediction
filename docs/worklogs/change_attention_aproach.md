# Change Attention Aproach (EOPKG)

Оновлено: 2026-04-08  
Статус: Proposal

## 1. Контекст проблеми
У поточних експериментах зміна `cross_attention_heads` (1, 2, 4, 8, 16, 32) дає майже нульовий приріст:
- F1/Loss відрізняються на рівні 3-го знаку після коми,
- час навчання змінюється незначно (порядку 2-3%).

Це означає, що крос-увага зараз або слабко впливає на фінальне рішення, або її сигнал "поглинається" іншими частинами моделі.

## 2. Як реалізовано зараз
Модель: `EOPKGGATv2` (`src/domain/models/eopkg_models.py`).

### 2.1 Поточний пайплайн (спрощено)
1. `Observed branch`:
- `x_cat/x_num -> concat -> GATv2(2 шари) -> pooling -> obs_context [B, H]`.
2. `Structural branch`:
- `struct_x` (або embedding класів) -> structural GNN -> `h_struct [C, Hs]`.
3. `Cross-attention`:
- `query = obs_context.unsqueeze(1)` має довжину 1 (`[B, 1, H]`),
- `key/value = h_struct` для всіх класів (`[B, C, H]`),
- отримуємо `context_struct`.
4. `Fusion`:
- `concat([obs_context, context_struct]) -> Linear + ReLU -> classifier`.

### 2.2 Що важливо
- Навіть з крос-увагою фінальна інтеграція все одно через простий `concat + MLP`.
- Один query-токен (довжина 1) обмежує виразність механізму уваги.
- За сильного observed-сигналу модель може опиратись переважно на `obs_context`.

## 3. Запропонована зміна
Додати керований конфігом режим злиття гілок:

- `model.fusion_mode: concat_mlp | gated | struct_pool_concat`

### 3.1 `concat_mlp` (поточний, без змін)
`concat(obs, struct_ctx) -> MLP -> logits`

### 3.2 `gated` (новий)
Навчуваний гейт керує внеском structural-контексту:
- `g = sigmoid(Wg [obs; struct])`
- `h = g * obs + (1 - g) * struct_proj`
- `h -> classifier`

Ефект: модель явно вчиться, коли довіряти структурі, а коли observed-гілці.

### 3.3 `struct_pool_concat` (новий, без cross-attention)
Без крос-уваги:
- `h_struct -> global pool (mean/attn-pool) -> s_ctx`
- `concat(obs, s_ctx) -> MLP -> logits`

Ефект: простіший і стабільніший шлях, близький до попереднього підходу "простого concat", який раніше давав ефект.

## 4. Чому очікується покращення
1. Менше залежності від `query_len=1` у cross-attention.
2. Явний контроль балансу observed/structural сигналів (`gated`).
3. Простий `struct_pool_concat` може краще переноситись на дані з паралельністю та нестабільним порядком завершення задач.
4. Легше діагностувати внесок структури:
- порівняння `fusion_mode` дає чистіший A/B тест, ніж лише зміна `cross_attention_heads`.

## 5. Плюси/мінуси зміни

### Плюси
- Краща інтерпретованість ролі structural branch.
- Можливість отримати реальний приріст без ускладнення attention-механіки.
- Менший ризик "псевдо-складності", коли параметр є, але практично не впливає.

### Мінуси
- Збільшується конфігураційний простір (`fusion_mode` + гіперпараметри).
- Потрібно перевалідувати сумісність з маскою (`mask_guided`) і метриками Stage2/Stage3.
- Можлива потреба в окремому тюнінгу для різних логів (RIO/Camunda).

## 6. Ризики та запобіжники
Ризики:
- деградація на датасетах, де cross-attention таки корисний;
- нестабільність при неправильній ініціалізації `gated`.

Запобіжники:
1. Залишити `concat_mlp` дефолтом (backward compatibility).
2. Додати тести форварду для всіх `fusion_mode`.
3. Порівнювати режими на одному seed-пакеті та однакових Stage2 метриках:
- `target_in_mask_rate`,
- `pred_in_mask_rate`,
- `strict_error_but_allowed_rate`,
- `test_oos`,
- `macro_f1`, `val_loss`.

## 7. Рішення щодо впровадження
Рекомендується впровадити поетапно:
1. Додати `fusion_mode` + `struct_pool_concat` (мінімальний ризик).
2. Додати `gated`.
3. Лише після цього вирішувати, чи ускладнювати cross-attention далі.

Критерій успіху:
- стабільне покращення `macro_f1` або `val_loss` без погіршення OOS/mask-метрик,
- або збереження якості при меншій обчислювальній складності.
